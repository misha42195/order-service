# Архитектура асинхронной обработки заказов для проекта SFMShop

В проекте реализована асинхронная модель обработки заказов.
Это позволяет API мгновенно отвечать пользователю,
тяжелые операции выполняются в фоне

## 1. Задачи для очереди

Задачи передаются в очередь в виде словарей, содержащих тип задачи по ключу task и необходимые параметры.

* **Отправка email-уведомлений send_email**:
    * *Зачем:* Взаимодействие с внешними SMTP-серверами может занимать несколько секунд.
    * *Данные:* order_id, user_email.
* **Обновление склада update_stock**:
    * *Зачем:* Гарантирует актуальность остатков товаров после совершения покупки без блокировки основного потока
      оформления заказа.
    * *Данные:* order_id.
* **Генерация отчетов generate_report**:
    * *Зачем:* Ресурсозатратная операция (формирование PDF или агрегация данных), которая не должна влиять на UX
      клиента.
    * *Данные:* order_id.

## 2. Надежность

Код реализован с учетом возможных ошибок(сетевые ошибки, падение БД):

* **Гарантии доставки**:
    * Используется механизм подтверждения задач task_done.
      Задача считается завершенной только после успешного выполнения функции-обработчика.
* **Повторная логика (Retry Logic)**:
    * Воркеры обернуты в блоки try-except.
    * При возникновении исключения задача увеличивает счетчик retry_count и возвращается в очередь (
      task_queue.put(message) для повторной попытки.
* **Атомарность**:
    * В API-роутере используется await db.commit(), и только после успешного сохранения заказа в БД задачи попадают в
      очередь. Это гарантирует, что воркер всегда найдет объект в базе по его ID.

## 3. Масштабирование

Архитектура позволяет системе расти вместе с количеством заказов:

* **Горизонтальное масштабирование**:
    * Поскольку воркеры потребляют задачи из одной общей очереди, мы можем запустить несколько идентичных
      процессов-потребителей на разных ядрах процессора или серверах.
* **Балансировка нагрузки**:
    * Используется стандартный механизм потокобезопасной очереди (`queue.Queue`), который распределяет задачи между
      первым освободившимся воркером.
