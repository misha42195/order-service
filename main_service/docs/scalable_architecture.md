# Масштабируемая архитектура для SFMShop

## Компоненты:

### 1. Load Balancer (nginx)
- Распределение нагрузки между API серверами
```pycon
Это процесс деления входящего потока пользователей между 
несколькими запущенными копиями моего приложения.
Зачем: Один сервер имеет предел мощности (CPU/RAM). 
Чтобы магазин не «упал» при наплыве 10 000 человек, 
ты запускаешь, например, 3 копии приложения, а 
Nginx распределяет запросы между ними (по очереди или по загрузке).
Результат: Система становится горизонтально масштабируемой.

```
- Health checks для проверки доступности серверов
```pycon
Это автоматический «мониторинг пульса» твоих API-серверов.
Зачем: Если один из трех серверов завис или на нем выключилось 
питание, Nginx поймет это (по отсутствию ответа или ошибке 500).
Результат: Nginx мгновенно исключает «больной» сервер из списка 
и отправляет всех людей на оставшиеся «здоровые» серверы. 
Покупатель даже не заметит, что внутри что-то сломалось.
```
- SSL termination
```pycon
Процесс расшифровки защищенного HTTPS-трафика прямо на входе.
Зачем: Шифрование данных (HTTPS) требует много ресурсов 
процессора. Вместо того чтобы каждый API-сервер 
тратил силы на расшифровку каждого клика, 
это делает один Nginx.
Результат: Внутри твоей защищенной сети 
(между Nginx и API) данные ходят по быстрому 
HTTP без шифрования. Это экономит ресурсы и 
упрощает управление сертификатами 
(ключ лежит только в одном месте — на Nginx).

```
- Rate limiting
```pycon
Установка лимитов на количество запросов от одного 
пользователя (IP-адреса) в единицу времени.
Зачем: Защищает от ботов, парсеров, которые 
пытаются выкачать все цены разом, или от 
злоумышленников (DDoS-атаки). Например, 
ты ставишь лимит: «не более 5 запросов в 
секунду с одного IP».
Результат: Если кто-то начнет закидывать 
сайт запросами слишком часто, Nginx выдаст 
ему ошибку 429 Too Many Requests, сохранив 
ресурсы сервера для реальных покупателей.
```


### 2. API Servers (FastAPI)
- 5+ экземпляров для обработки запросов
```pycon
Вместо одного большого сервера запускается 
минимум 5 независимых копий (инстансов) приложения.
Зачем: Это исключает появление «единой точки отказа». 
Если один процесс зависнет из-за ошибки в коде, 
остальные 4 продолжат обслуживать покупателей. 
Это обеспечивает высокую доступность (High Availability).
```
- Горизонтальное масштабирование
```pycon
Это стратегия увеличения мощности системы путем 
добавления новых серверов в кластер, а не покупкой 
более дорогого «железа».
Зачем: У одного сервера есть физический предел. 
При горизонтальном подходе ты просто доставляешь 
рядом новые «коробки» (или Docker-контейнеры), 
и Nginx начинает делить трафик между ними. 
Это позволяет системе расти бесконечно.
```
- Автоматическое масштабирование при росте нагрузки
```pycon
Механизм, который сам нанимает «дополнительных сотрудников» 
(серверы), когда в магазине ажиотаж.
Зачем: Если в «Черную пятницу» нагрузка 
на CPU подпрыгивает выше 70%, система 
(например, Kubernetes или AWS Auto Scaling) 
сама запускает еще 5–10 экземпляров FastAPI. 
Когда люди ложатся спать и нагрузка падает — лишние 
серверы выключаются, экономя твои деньги.
```
- Stateless архитектура
```text
Принцип разработки, при котором сервер не хранит 
внутри своей памяти данные о конкретном пользователе 
(например, кто залогинен или что лежит в корзине).
Зачем: Это критически важно для масштабирования. 
Запрос пользователя №1 может попасть на Сервер А, 
а следующий его клик — на Сервер Б. Если данные 
хранятся в памяти сервера, Сервер Б «забудет» пользователя.
Результат: Все данные о состоянии (сессии, корзины) 
выносятся во внешнее быстрое хранилище (Redis), 
что делает API-серверы взаимозаменяемыми «роботами».
```

### 3. Redis Cache
- Кэширование результатов запросов (TTL 1 час)
```text
Временное хранение «тяжелых» данных из базы данных 
(например, список товаров или дерево категорий) 
в оперативной памяти.
Зачем: Запрос к базе данных (PostgreSQL) 
может занимать 100–200 мс, а чтение из 
Redis — менее 5 мс. TTL (Time To Live) 1 час означает, 
что данные в кэше обновятся автоматически через 60 минут.
Результат: Сайт «летает», а основная база данных не 
перегревается от одинаковых запросов.
```
- Сессии пользователей
```text
Хранение данных о том, какой пользователь залогинился 
и что он делает на сайте прямо сейчас.
Зачем: Поскольку наши API-серверы Stateless 
(ничего не помнят), им нужно место, где они будут 
быстро проверять: «А этот токен вообще верный?».
Результат: Пользователь не «разлогинивается» при 
переходе между страницами, даже если его запросы 
попадают на разные серверы API.
```
- Rate limiting
```text
Технический счетчик запросов для каждого 
пользователя в реальном времени.
Зачем: Nginx может ограничивать общие лимиты, 
но Redis позволяет делать это гибко 
на уровне приложения. Например: «запретить 
добавлять в корзину чаще 10 раз в минуту».
Результат: Защита от скриптов-ботов, которые 
могут скупить все товары за секунду или перегрузить поиск.
```
- Кластеризация для высокой доступности
```text
Объединение нескольких серверов Redis в единую сеть, 
где данные дублируются.
Зачем: Если у вас один Redis и он «упал» — сайт 
перестанет пускать пользователей (так как сессии 
недоступны). Кластеризация позволяет системе работать, 
даже если один из серверов кэша выйдет из строя.
Результат: Гарантия работы магазина 24/7 без потерь 
данных о корзинах и сессиях.
```

### 4. PostgreSQL
- Master для записи
```text
Выделенный основной сервер базы данных, 
который обрабатывает все изменяющие данные операции 
(INSERT, UPDATE, DELETE).
Зачем: В любой системе должен быть «источник истины». 
Когда пользователь нажимает «Оформить заказ», 
данные идут строго на Master-сервер, чтобы избежать 
конфликтов и путаницы в номерах заказов.
```
- 2+ Replicas для чтения
```text
Копии основной базы данных, предназначенные только 
для выполнения запросов на получение данных (SELECT).
Зачем: В интернет-магазине просмотров товаров в 
десятки раз больше, чем реальных покупок. 
Мы разгружаем основной сервер, отправляя всех, 
кто просто «смотрит витрину», на реплики.
Результат: Основная база всегда свободна 
для принятия денег (заказов), а каталог 
открывается мгновенно.
```

- Репликация в реальном времени(Streaming Replication)
```text
Процесс мгновенной передачи каждого изменения с 
Master-сервера на все реплики.
Зачем: Как только администратор изменил цену 
товара на Master, через доли секунды эта цена 
должна обновиться на всех репликах.
Результат: Пользователи всегда видят актуальные 
остатки товаров и цены, на какой бы сервер базы 
их ни забросил балансировщик.
```
- Автоматический failover
```text
Механизм, который в случае поломки 
Master-сервера автоматически назначает 
одну из реплик «новым главным».
Зачем: Если сервер с основной базой «сгорит», 
магазин не должен закрыться. Система мониторинга 
заметит сбой и за секунды перенаправит поток записи 
на здоровую реплику.
Результат: Непрерывная работа бизнеса 
(High Availability) и защита от потери данных о 
последних заказах.
```

### 5. RabbitMQ
- Очереди для асинхронной обработки
```text
Механизм «отложенного» выполнения задач. 
Когда пользователь нажимает кнопку «Купить», 
сервер не заставляет его ждать, пока пройдут 
все проверки, а просто кидает сообщение в очередь 
и говорит: «Принято!».
Зачем: Чтобы пользователю не пришлось смотреть 
на крутящийся лоадер 10–20 секунд. Сложные задачи 
выполняются фоновыми рабочими (Workers) отдельно 
от основного сайта.
Результат: Мгновенный отклик интерфейса даже при 
высокой нагрузке.
```
- Обработка заказов, платежей, уведомлений
```text
Разделение критически важных бизнес-процессов 
на независимые этапы.
Зачем: Оплата может занять время, отправка Email 
через сторонний сервис может «тормозить», 
а генерация чека — нагружать процессор. 
RabbitMQ гарантирует, что каждая задача 
(платеж, письмо, лог) будет выполнена по порядку.
Результат: Если сервис рассылки писем временно упал, 
очередь просто накопит письма и отправит их позже, 
когда сервис оживет. Ни одно уведомление не потеряется.
```
- Dead Letter Queue для обработки ошибок
```text
Специальный «карантин» для задач, 
которые не удалось выполнить после нескольких попыток.
Зачем: Если какой-то заказ вызывает ошибку 
(например, неверный формат данных), 
он не должен блокировать всю очередь 
и «вешать» систему. Он переносится в Dead Letter Queue.
Результат: Разработчики могут позже зайти в эту очередь,
разобраться в причинах ошибки и перезапустить 
конкретную задачу вручную, не ломая работу всего магазина.
```

### 6. CDN
- Кэширование статических ресурсов
```text
Копирование «тяжелых» и редко меняющихся файлов (фотографии товаров,
видеообзоры, JS-скрипты, CSS-стили) на промежуточные сервера-посредники.
Зачем: Чтобы при каждом просмотре кроссовок ваш основной сервер API 
не тратил ресурсы на отдачу картинки весом 2 МБ. CDN сделает это 
один раз и будет раздавать копию.
Результат: Основной сервер занимается только заказами, а картинки 
отдаются мгновенно из кэша CDN.
```
- Распределение по регионам
```text
Размещение серверов CDN в разных географических точках 
(например, в Москве, Новосибирске, Берлине, Нью-Йорке).
Зачем: Скорость света ограничена. Если сервер в Москве, 
а покупатель во Владивостоке, картинка будет грузиться долго. 
CDN найдет ближайший к пользователю сервер («Edge») и отдаст файл оттуда.
Результат: Минимальный пинг и моментальная загрузка 
сайта вне зависимости от того, где находится покупатель.
```
- Снижение нагрузки на серверы
```text
Перехват до 90% всего трафика на сторону сети CDN.
Зачем: Без CDN при наплыве 100 000 пользователей ваш Nginx 
может захлебнуться от количества запросов за картинками. 
CDN забирает этот удар на себя.
Результат: Радикальное снижение затрат на трафик основного 
сервера и защита от падения системы при вирусном росте популярности товара.

```

## Масштабирование:

### Горизонтальное:
- API серверы: добавлять экземпляры при RPS > 4000
```text
Автоматическое увеличение количества запущенных 
FastAPI-контейнеров при достижении порога в 4000 запросов в секунду.
Зачем: При такой нагрузке процессорные мощности текущих 
серверов могут исчерпаться, что приведет к росту задержек (Latency).
Результат: Система сохраняет стабильную скорость отклика 
(например, < 200 мс) даже во время резких пиков трафика, 
просто «размазывая» запросы по новым серверам.
```
- Redis: кластеризация для распределения нагрузки
```text
Переход от одного сервера к группе узлов (Redis Cluster), 
где данные (сессии, кэш) распределяются по частям (шардам).
Зачем: Один сервер Redis ограничен объемом оперативной памяти и 
пропускной способностью одного сетевого порта. 
Кластер позволяет объединить память 3, 5 или 10 серверов в одну сверхбыструю базу.
Результат: Возможность хранить миллионы активных корзин 
и сессий одновременно без потери скорости доступа.
```
- PostgreSQL: добавлять replicas для чтения
```text
Создание дополнительных копий базы данных, которые 
только отдают информацию пользователям.
Зачем: В интернет-магазине на один созданный заказ 
приходится тысячи просмотров страниц товаров. 
Добавление реплик позволяет бесконечно масштабировать «витрину».
Результат: Основной сервер (Master) всегда свободен 
для быстрой записи новых заказов, а любая нагрузка на 
поиск и просмотр товаров гасится за счет новых реплик.
```

### Вертикальное:
- PostgreSQL Master: увеличение ресурсов при росте данных
```text
Наращивание мощностей (CPU, RAM, быстрые NVMe-диски) 
на основном сервере базы данных.
Зачем: В отличие от API-серверов, базу данных, 
принимающую заказы (Master), сложнее разделить на части. 
Чтобы она справлялась с растущим потоком транзакций и 
сложными запросами на запись, ей нужно больше «железа».
Результат: Гарантированная скорость обработки платежей и 
создания заказов без задержек при увеличении объема базы до терабайт.
```
- Redis: увеличение памяти для кэша
```text
Добавление оперативной памяти (RAM) на серверы кэширования.
Зачем: Redis хранит всё в оперативной памяти. 
Если память закончится, он начнет удалять старые данные 
(по алгоритму LRU), что заставит приложение чаще ходить 
в медленную базу данных.
Результат: Возможность хранить в быстром доступе еще 
больше популярных товаров, сессий и корзин, сохраняя 
моментальный отклик сайта для всех пользователей.

```

### Автоматическое:
- Kubernetes HPA для автоматического масштабирования API
```text
Это использование встроенного механизма Horizontal Pod Autoscaler. 
Он автоматически изменяет количество реплик (подов) вашего 
API в зависимости от нагрузки. Если запросов становится много, 
HPA создает новые поды; когда нагрузка спадает — удаляет лишние, 
чтобы не тратить ресурсы [1, 3].
```
- Мониторинг метрик (CPU, memory, RPS)
```text
Это сбор и анализ показателей работы приложения в реальном времени. 
CPU и Memory показывают потребление «железа», а RPS (запросы в секунду) 
отражает фактическую нагрузку на сервис. 
Эти данные служат триггером для принятия решения о масштабировании [2, 3].
```
- Автоматическое добавление/удаление экземпляров
```text
Непосредственный результат работы HPA. 
Система поддерживает заданный баланс: если текущие экземпляры 
загружены выше лимита, Kubernetes добавляет новые. 
Как только потребление ресурсов снижается, 
система удаляет избыточные экземпляры, оптимизируя затраты [1, 4].
```

## Отказоустойчивость:

### Health Checks:
- Проверка доступности API каждые 10 секунд
```text
Это настройка интервала опроса (periodSeconds). 
Раз в 10 секунд система отправляет запрос 
(например, на эндпоинт /health) к каждому экземпляру API. 
Это позволяет оперативно обнаружить «зависший» или упавший процесс, 
не дожидаясь жалоб от пользователей.
```
- Автоматическое исключение недоступных серверов
```text
Если проверка (Health Check) провалена несколько раз подряд, 
сервер помечается как «нездоровый» (Unhealthy). 
Система мгновенно перестает направлять на него трафик. 
Пользователи не получают ошибок, так как запросы уходят только на рабочие экземпляры.
```
- Автоматическое восстановление при доступности
```text
Система не удаляет сервер навсегда, а продолжает его опрашивать. 
Как только API снова начинает отвечать корректно 
(сервер «прогрелся» или восстановил связь с базой), 
он автоматически возвращается в ротацию и снова начинает принимать пользовательский трафик.

```

### Резервное копирование:
- Ежедневное резервное копирование БД
```text
Это настройка автоматического расписания (cron job), 
которое раз в сутки делает полный слепок (дамп) базы данных. 
Это гарантирует, что в случае критического сбоя 
максимальная потеря данных составит не более 24 часов.
```
- Хранение бэкапов 30 дней
```text
Это политика ротации (retention policy). Система автоматически удаляет 
копии старше одного месяца, чтобы не переплачивать за дисковое пространство. 
При этом у вас всегда есть «окно» в 30 дней, чтобы откатиться к 
состоянию на любую дату в этом промежутке.
```
- Тестирование восстановления раз в неделю
```text
Это критически важный процесс проверки валидности бэкапов. 
Раз в неделю система (или администратор) берет одну из копий и 
разворачивает её на тестовом стенде. Это подтверждает, 
что файлы не битые, пароли подходят, а данные реально пригодны 
для восстановления в экстренной ситуации.
```

### Мониторинг:
- Prometheus для сбора метрик
```text
Это база данных временных рядов, которая выступает «мозгом» мониторинга. 
Она регулярно опрашивает ваши сервисы и Kubernetes, 
собирая числовые данные (метрики) о загрузке, ошибках и скорости ответов.
```
- Grafana для визуализации
```text
Это графический интерфейс, который превращает «сухие» цифры из 
Prometheus в наглядные дашборды с графиками. С её помощью можно 
за секунды увидеть общую картину состояния системы, не заглядывая в логи.
```
- Алерты при превышении порогов
```text
Это система мгновенных уведомлений (Alertmanager). 
Если какая-то метрика (например, загрузка CPU или процент ошибок 5xx) 
выходит за допустимые границы, система автоматически отправляет сообщение в Telegram, 
Slack или на почту, чтобы инженеры узнали о проблеме раньше пользователей.
```

## Метрики и SLA:

### Целевые метрики:
- RPS: 5000+ запросов в секунду
```text
Это целевая пропускная способность (Throughput). 
Система проектируется и тестируется так, чтобы стабильно обрабатывать 
поток из 5000 запросов каждую секунду без деградации производительности и отказов.
```
- Latency: < 100ms для 95% запросов
```text
Это показатель задержки (перцентиль p95). 
Он означает, что для подавляющего большинства пользователей 
(95 из 100) ответ от API приходит быстрее, чем за 0.1 секунды. 
Это гарантирует высокую скорость работы интерфейса.
```
- Availability: 99.9% (uptime)
```text
Это коэффициент доступности («три девятки»). 
В пересчете на время это означает, что суммарный простой системы 
(даунтайм) не должен превышать 8.7 часов в год или около 43 минут в месяц.
```
- Error rate: < 0.1%
```text
Это допустимый порог ошибок. Показатель означает, 
что на каждые 1000 запросов допускается не более одного 
неудачного ответа по вине сервера (ошибки 5xx). 
Это критично для поддержания доверия пользователей к сервису.
```

### План масштабирования:
- При RPS > 4000: добавить 2 API сервера
```text
Это правило превентивного (упреждающего) расширения. 
Когда нагрузка приближается к целевому пределу в 5000 RPS, 
система заранее увеличивает вычислительную мощность, чтобы 
сохранить запас прочности и избежать деградации сервиса.
```
- При latency > 150ms: проверить кэш и оптимизировать запросы
```text
Это триггер для технического анализа. Если задержка превышает норму, 
проблема часто кроется не в количестве серверов, а в медленной 
базе данных или неэффективном кэшировании. Это сигнал инженерам 
проверить «узкие места» в коде или инфраструктуре.
```
- При ошибках > 0.5%: проверить здоровье серверов
```text
Это экстренный индикатор сбоя. Превышение порога в 0.5% ошибок означает, 
что проблема носит системный характер (например, отвал связи с БД или битый деплой). 
Требуется немедленная диагностика логов и статусов (Health Checks) всех узлов.
```
- При использовании CPU > 80%: добавить экземпляры
```text
Это классическое правило горизонтального масштабирования. 
Высокая загрузка процессора снижает скорость обработки запросов. 
Автоматика вводит в строй новые поды, чтобы распределить 
вычислительную нагрузку и вернуть загрузку каждого ядра в безопасную зону (обычно 50–70%).
```
