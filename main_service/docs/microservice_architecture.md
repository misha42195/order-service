# Структура микросервиса для обработки платежей:

## pyment-service

### Структура:

- Корень директории микросервиса **./payment-service/src**

1) src/api(main.py,routes/pyments.py) - слой представления обработка маршрутов
2) src/servieces/pyment_service.py - бизенес логика работы с данными(crud)(взаимодействие с бд платежек)
3) scr/database/(connection.py, models.py) - работа с базой данных(модели орм, подключение к бд)
4) src/schemas/pyments.py - схемы  
5) src/core/config.py - настройки конфигурации

- payment-service/ tests - тесты
- dockerfile - докер файл для развертывания микросервиса
- requirements.txt - установленные библиотеки
**Ответственность микросервиса** -
- Создание платежки при оформлении заказа
- Хранение статуса платежей
- Возврат средств
- 
### Endpoints:

POST api/v1/payments - после создания заказа
формируем/создаем новую платежку
GET api/v1/payments/{id}) - получить платежку по указанному id


в микросервисе который отвечает за формирование заказов
описываем взаимодействие с микросервисом создания платежек.
То есть у нас будет файл в котором будет реализован механизм
взаимодействия с данными платежек, а именно будут запросы к
микросервису платежек.

**Как order-service вызывает payment-service** -
в order-service описывается работа работа с заказами
а так же включается свойство(self.pyments) - которое ссылается
на объект PymentClient в котором описаны методы для взаимодействи
с сервисом платежек. В методе PymentClient указыается путь с
хостом и портом для запросов к микросервису платежек.
**Какие данные передаются**- при создании заказа создаются
поля для заказа и определенные данные передаются в метод который
реализован у объекта PymentsClient. Это order_id заказа и сумма.
**Как обрабатываются ошибки** -
1) После создания заказа оптравляем запрос на сервис платежек 
а сервис недоступен. 
2) Ошибка возвращется в сревис заказа и заказ помечается как отмененный
   (так как платежка не сформирована и не оплачена) клиенту показываем 
что заказ отменен
**Опиши развертывание** - 
Для каждого микросервиса обязательно используются отдельные 
порты, база данных, свои настройки для docker, свои библиотеки 
и конечно же пишутся свои тесты. 
**Как взаимодействуют** - 
1) клиент обращется к микросервису заказа,
2) сервис заказа обрабатывает данные и обращется к сервису платежек
3) сервис платежек принимает данные и обрабатывает 
4) передает данные или ошибку в сервис заказа
5) данные передаются клиенту
